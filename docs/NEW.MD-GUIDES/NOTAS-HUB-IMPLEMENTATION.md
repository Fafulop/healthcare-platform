# Notas Hub — Implementation Guide

**Feature:** Standalone note-taking workspace for doctors
**Route:** `/dashboard/notas`
**Status:** IN PROGRESS — Steps 1–10 complete, Railway migration pending

---

## Overview

The Notas Hub is a dedicated space where doctors write and organize notes. Notes are independent — they are not part of any patient record, encounter, or task. From a note, a doctor can optionally create a task in `/dashboard/pendientes`; once created, that task is fully detached from the note (no sync, no reference).

Patient-specific notes (inside each patient's expediente) are a **separate future feature** — do not conflate them with this implementation.

---

## Core Features

- Write notes as plain text
- Dictate via OpenAI Whisper (press to start, press again to stop → appends transcription)
- Organize with a 2-level label system: **Tema** → **Subtema** (both optional)
- Labels are created on the fly via free-text combobox (no separate management UI)
- Browse/filter notes via a sidebar index tree
- Full-text search across note content + tema/subtema names (client-side)
- Edit notes at any time
- Create a task from a note (fire-and-forget — task becomes independent)
- Delete notes

## Database Schema

### 3 new tables in `medical_records` schema

#### `note_temas`
Per-doctor label taxonomy — top level.

| Column | Type | Notes |
|---|---|---|
| `id` | TEXT PK | cuid() — generated by Prisma |
| `doctor_id` | TEXT FK | → `public.doctors(id)` ON DELETE CASCADE |
| `name` | TEXT | Free text, unique per doctor |
| `created_at` | TIMESTAMP(3) | |

Unique constraint: `(doctor_id, name)`

#### `note_subtemas`
Second-level labels, always belong to a tema.

| Column | Type | Notes |
|---|---|---|
| `id` | TEXT PK | cuid() — generated by Prisma |
| `tema_id` | TEXT FK | → `note_temas(id)` ON DELETE CASCADE |
| `doctor_id` | TEXT FK | → `public.doctors(id)` ON DELETE CASCADE |
| `name` | TEXT | Free text, unique per tema |
| `created_at` | TIMESTAMP(3) | |

Unique constraint: `(tema_id, name)`

#### `doctor_notes`
The notes themselves.

| Column | Type | Notes |
|---|---|---|
| `id` | TEXT PK | cuid() — generated by Prisma |
| `doctor_id` | TEXT FK | → `public.doctors(id)` ON DELETE CASCADE |
| `content` | TEXT | Plain text, default `''` |
| `tema_id` | TEXT FK? | → `note_temas(id)` ON DELETE SET NULL, nullable |
| `subtema_id` | TEXT FK? | → `note_subtemas(id)` ON DELETE SET NULL, nullable |
| `created_at` | TIMESTAMP(3) | |
| `updated_at` | TIMESTAMP(3) | Managed by Prisma `@updatedAt` — no SQL DEFAULT needed |

Indexes: `doctor_id`, `tema_id`

```prisma
model NoteTema {
  id        String        @id @default(cuid())
  doctorId  String        @map("doctor_id")
  name      String
  createdAt DateTime      @default(now()) @map("created_at")

  doctor    Doctor        @relation(fields: [doctorId], references: [id], onDelete: Cascade)
  subtemas  NoteSubtema[]
  notes     DoctorNote[]

  @@unique([doctorId, name])
  @@index([doctorId])
  @@map("note_temas")
  @@schema("medical_records")
}

model NoteSubtema {
  id        String       @id @default(cuid())
  temaId    String       @map("tema_id")
  doctorId  String       @map("doctor_id")
  name      String
  createdAt DateTime     @default(now()) @map("created_at")

  tema      NoteTema     @relation(fields: [temaId], references: [id], onDelete: Cascade)
  doctor    Doctor       @relation(fields: [doctorId], references: [id], onDelete: Cascade)
  notes     DoctorNote[]

  @@unique([temaId, name])
  @@index([temaId])
  @@index([doctorId])
  @@map("note_subtemas")
  @@schema("medical_records")
}

model DoctorNote {
  id        String       @id @default(cuid())
  doctorId  String       @map("doctor_id")
  content   String       @default("")
  temaId    String?      @map("tema_id")
  subtemaId String?      @map("subtema_id")
  createdAt DateTime     @default(now()) @map("created_at")
  updatedAt DateTime     @updatedAt @map("updated_at")

  doctor    Doctor       @relation(fields: [doctorId], references: [id], onDelete: Cascade)
  tema      NoteTema?    @relation(fields: [temaId], references: [id], onDelete: SetNull)
  subtema   NoteSubtema? @relation(fields: [subtemaId], references: [id], onDelete: SetNull)

  @@index([doctorId])
  @@index([temaId])
  @@map("doctor_notes")
  @@schema("medical_records")
}
```

**Doctor model reverse relations added:**
```prisma
doctorNotes   DoctorNote[]
noteTemas     NoteTema[]
noteSubtemas  NoteSubtema[]
```

## ✅ Step 2 — API Routes — DONE

**Files created:**
- `apps/doctor/src/app/api/notes/route.ts` ✅
- `apps/doctor/src/app/api/notes/[id]/route.ts` ✅
- `apps/doctor/src/app/api/notes/temas/route.ts` ✅

**Key implementation notes:**
- Prisma generates IDs via `@default(cuid())` — do NOT pass `id` manually in create calls
- Temas/subtemas are upserted inline during note create/update — no separate create endpoint
- Prisma unique key names: `doctorId_name` for NoteTema, `temaId_name` for NoteSubtema
- PUT always clears tema/subtema if not included in body — frontend must send current values
- Ownership checked via `findFirst({ where: { id, doctorId } })` — returns 404 for wrong doctor (security)
- Auth via `requireDoctorAuth(request)` from `@/lib/medical-auth`
- Error handling via `handleApiError` from `@/lib/api-error-handler`

**Endpoints:**
```
GET  /api/notes?q=        list notes, optional search (server-side, insensitive)
POST /api/notes           create note + upsert tema/subtema
GET  /api/notes/[id]      single note
PUT  /api/notes/[id]      update note + upsert tema/subtema
DELETE /api/notes/[id]    delete note
GET  /api/notes/temas     all temas with subtemas[] + _count.notes
```

**Response shape:**
```json
{ "success": true, "data": { ...note, "tema": { "id": "...", "name": "..." }, "subtema": null } }
```

## ✅ Step 3 — Hook: `useNotesPage.ts` — DONE

**File:** `apps/doctor/src/app/dashboard/notas/_hooks/useNotesPage.ts` ✅

**Exported types:** `Note`, `NoteLabel`, `NoteTema`, `NoteSubtema`

**Key implementation notes:**
- `filteredNotes` is computed inline (not memoized) — fine for typical note counts
- Sidebar filter uses sentinel `'__none__'` as `filterTemaId` to match notes with no tema
- `setFilter(null)` clears all filters (shows all notes)
- `setFilter(temaId)` filters by tema; `setFilter(temaId, subtemaId)` filters by subtema
- `handleTemaChange` auto-clears `editorSubtemaName` when tema changes
- `saveNote` re-fetches temas after save to keep sidebar counts in sync
- `deleteNote` closes editor if the deleted note was open
- Whisper: field name must be `'audio'` (not `'file'`) when appending to FormData
- Whisper response path: `data.data.transcript` (not `data.text`)
- Whisper error path: `data.error?.message`
- `recording` state goes false immediately on stop click; `transcribing` takes over while Whisper processes
- `mediaRecorderRef.current?.stream` is valid Web API — `MediaRecorder` exposes the stream it was created with

**Hook returns:**
```ts
// Data
notes, temas, loadingNotes
// Derived
filteredNotes, selectedNote
// Sidebar
filterTemaId, filterSubtemaId, search, setSearch, setFilter
// Editor state
isNewNote, editorContent, editorTemaName, editorSubtemaName, isDirty, saving
handleContentChange, handleTemaChange, handleSubtemaChange
// Editor actions
selectNote, newNote, closeEditor, saveNote, deleteNote
// Whisper
recording, transcribing, toggleRecording
// Task modal
showTaskModal, openTaskModal, closeTaskModal
```

## ✅ Step 9 — CreateTaskFromNoteModal — DONE

**File:** `_components/CreateTaskFromNoteModal.tsx`

**Props:**
```ts
{
  isOpen: boolean;
  onClose: () => void;
  noteContent: string;  // to pre-fill title
}
```

**Form fields:**
| Field | Default | Required |
|---|---|---|
| Título | First line of `noteContent`, editable | Yes |
| Fecha | Today | Yes |
| Hora | (empty) | Yes |
| Categoría | OTRO | Yes |
| Prioridad | MEDIA | Yes |

**On submit:** POST to `/api/medical-records/tasks`
```json
{
  "title": "...",
  "dueDate": "2026-03-25",
  "startTime": "10:00",
  "endTime": "10:30",
  "category": "SEGUIMIENTO",
  "priority": "MEDIA"
}
```
- Time is optional; if provided, `startTime`+`endTime` (= startTime+30min) are sent — API requires both or neither, in 30-min increments
- Tasks API returns `{ data: task }` (not `{ success: true }`) with status 201
- No note reference stored. On success: `toast.success('Tarea creada')` + `onClose()`.

**Task categories enum:** `SEGUIMIENTO | ADMINISTRATIVO | LABORATORIO | RECETA | REFERENCIA | PERSONAL | OTRO`
**Task priorities enum:** `ALTA | MEDIA | BAJA`

## Key Design Decisions

| Decision | Choice | Reason |
|---|---|---|
| Tema creation | Free-text combobox, upsert on save | Zero friction, no separate management UI |
| Patient link | None — separate future feature | Keeps scope clean |
| Task reference | No back-reference from task to note | Tasks are independent; avoids sync complexity |
| Text format | Plain text only | Works perfectly with Whisper append |
| Search | Client-side filter on loaded notes | Doctor note counts won't be large enough to need server-side |
| Whisper UX | Press to start / press again to stop → appends | Confirmed pattern |
| Orphaned temas | Kept (not auto-deleted) | Simpler; doctor reuses labels |
| Auto-save | No — explicit Guardar button | Avoids partial saves during Whisper or mid-thought |
| Task modal dueTime | Single time field (not start+end) | Tasks from notes are reminders, not time-blocked events |
